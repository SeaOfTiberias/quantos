# S8-3 Momentum Backtest — Methodology, Pre-Committed 2026-07-19 Before Any Result Exists

Fable's post-Darvas review ranked 52-week-high relative-strength momentum as
the strongest remaining candidate — it's the actual documented cross-
sectional anomaly (George & Hwang 2004: stocks near their 52-week high tend
to keep outperforming; investors underreact to news that pushes a price to
a new high), unlike Darvas's borrowed single-name-breakout citation. This
doc fixes every methodology choice before `scripts/backtest_rs_momentum.py`
runs, so nothing here gets tuned after seeing results — same discipline as
`docs/S7_3_BACKTEST_SAMPLE.md`.

**Do not edit this file after the backtest has run.** If a parameter turns
out wrong, that's a finding for a NEXT pre-registered run, not a retroactive
edit to this one.

## Universe

`agent/universe_nifty500.txt` in full (500 symbols) — the same committed
list Stage A and regime breadth already use. No sampling: unlike S7-3
(single-symbol TradingView backtests, sampling needed to keep manual clicks
bounded), this is a Python harness that ranks the whole universe at each
rebalance, so there's no reason to sample a subset.

## Ranking metric

`nearness_score = daily_close / rolling_252_trading_day_high`, computed
from DAILY highs (not weekly closes) — a "52-week high" is the highest
price reached at any point in the past year, not just the highest Friday
close, so the rolling high must come from daily data even though rebalancing
is weekly. Higher score (closer to 1.0, i.e. closer to the 52-week high) =
stronger momentum candidate. A symbol needs >=252 daily bars of history
before it's eligible for ranking (excludes recent IPOs until they accrue
enough — same warmup logic already used for regime/EMA calculations
elsewhere in this project, not a special case invented for this backtest).

## Portfolio construction

**Top 20 symbols by nearness_score, rebalanced weekly.** 20 (4% of the
500-symbol universe) is a moderate, tradeable concentration — closer to a
practitioner portfolio than the academic literature's full decile/quintile
sort, chosen because a 50-name decile portfolio isn't something a retail
account actually runs. This is a judgment call made BEFORE running, not
tuned after seeing which N looks best.

Rebalance date = the last NIFTY trading day of each ISO calendar week
(Mon-Sun) within the backtest window, using NIFTY's own daily calendar as
the master date list (same anchoring pattern S8-1 used for its regime
replay) so every symbol rebalances against a common calendar rather than
its own individually-gapped trading days.

## Entry/exit mechanics

Pure rotation, no smoothing: a symbol generates a BUY the week it FIRST
enters the top 20, and a SELL the week it FIRST leaves the top 20 (or at
the last rebalance date if still held when the backtest window ends).
Holding period is whatever that streak turns out to be — not fixed. No
anti-chattering buffer at the rank-20 boundary: if the strategy trades a
lot because names hover right at the cutoff, that's a real, honest cost to
measure, not something to paper over before the first result exists
(mirrors Darvas's own actual cause of death — turnover eating a thin edge).

## Cost model

New `CostModel` instance with delivery-style rates (this is a weekly hold,
not intraday) per `core/risk/costs.py`'s own docstring guidance:
- **Brokerage: Rs0 flat.** Discount brokers (Fyers included) typically
  charge zero brokerage on equity delivery trades — this is a real,
  well-known difference from the intraday stack, not an oversight.
- **STT: approximated at 0.2% applied to the sell leg only** (double the
  real 0.1%-both-legs delivery rate). `CostModel.round_trip()` structurally
  applies STT to the sell leg only (built for the intraday sell-side-only
  rule) — doubling it approximates the real both-legs charge under the
  assumption that buy and sell turnover are usually close in size. Flagged
  as an approximation, not a structural fix to the shared CostModel class.
- **Stamp duty: 0.015% on the buy leg** (delivery rate, vs intraday's
  0.003%) — `CostModel` already applies stamp on the buy leg only, which
  matches the real rule structurally, no approximation needed here.
- **Exchange txn / SEBI turnover fee: same rates as intraday** — these
  apply uniformly regardless of product type.
- **Slippage: 10bps/leg default**, sensitivity-checked at 0bps — lower than
  S7-3's Darvas assumption (15-40bps) because entering/exiting a Nifty 500
  name near a weekly close is far less urgent than chasing an intraday
  breakout fill.

## Backtest window

**3 years of daily history** — long enough for a meaningful number of
weekly rebalances (~150) with several hundred symbol-months of ranking
data, short enough to keep the ~500-symbol chunked fetch runtime reasonable
(same throttled pattern as S8-1's breadth fetch, which took ~5 minutes for
100 symbols over 6 years — 500 symbols over 3 years is a similar order of
magnitude, run in the background).

## Pass/fail bar

**Same bar used throughout this project:**
`BacktestMetrics.has_positive_edge` (`core/backtest/parser.py`) — profit
factor > 1.0 AND Sharpe > 0.5, computed on the UNFILTERED result (S8-1
already showed the regime classifier doesn't reliably separate outcomes;
if a regime-split variant is computed at all, it's informational only, not
the basis for go/no-go — matching what S8-4 already did).

## What would make this backtest untrustworthy after the fact

- Changing N, the rebalance cadence, or the ranking window after seeing a
  weak or strong result.
- Adding an anti-chattering buffer only after seeing high turnover hurt
  the number.
- Reporting a regime-filtered result as the headline instead of unfiltered.

If any of the above turns out to matter, it becomes a new, separately
pre-registered run — not a retroactive edit here.
